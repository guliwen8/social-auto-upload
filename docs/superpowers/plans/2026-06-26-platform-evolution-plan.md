# 多人多自媒体自动运营平台演进计划

## 目标

把当前以 `CLI + uploader + skills` 为主线的 `social-auto-upload`，逐步演进成一套可供多人使用、支持多账号、多平台、带 AI 能力与风控治理的自媒体自动运营平台。

这次规划遵循几个前提：

- 充分复用现有成熟能力，尤其是 `uploader/*`、`sau_cli.py`、`skills/*`
- 旧的 `Flask + Vue` Web 路径只作为参考，不再作为未来主干
- 先满足你个人可用，再扩展到团队协作，最后具备产品化能力
- AI 必须接入，但默认先做人机协同，而不是一开始就全自动
- 平台风控要作为底层约束，而不是后补功能

## 当前判断

现有仓库最有价值的资产，不是旧 Web，而是已经成型的三层主线：

- `uploader/*`：各平台自动化适配器，沉淀了真实平台经验
- `sau_cli.py`：统一入口，已经具备平台引擎雏形
- `skills/*`：天然适合作为未来 AI/Agent 接入层

现有历史 Web 路径包括：

- `sau_backend.py`
- `sau_backend/`
- `sau_frontend/`
- `myUtils/*`

这些代码可以保留作参考和迁移，但不适合继续承接新平台需求。原因很简单：它们是单机式、弱隔离、弱治理的原型，不具备多人、多租户、审计、任务编排和凭证安全的基础。

## 总体方向

未来最合适的路线不是“把旧 Web 升级成 SaaS”，而是：

1. 保留并平台化现有 `CLI + uploader + skills`
2. 冻结旧 Web 路径
3. 新建控制面与执行面
4. 让 AI、风控、调度、审计都建立在统一内核之上

一句话概括：

旧 Web 不升级，CLI 不废弃，uploader 不重写，skills 不丢；新平台在其上增加控制层、调度层、治理层和 AI 层。

## 演进原则

### 保留什么

- `uploader/*`
- `sau_cli.py`
- `skills/*`
- `docs/CLI.md`、`docs/install.md` 这类主线文档
- 已有测试思路和平台约束经验

### 放弃什么

- 不再把 `sau_backend.py` 作为未来主后端
- 不再把 `sau_frontend/` 作为未来主前端
- 不再沿用 `myUtils/*` 这套旧调用链做增量开发
- 不再围绕本地文件散落式 cookie 存储继续扩展多人能力

### 优先级原则

- 先把执行引擎平台化，再做 AI 自动运营
- 先把四个主线平台做深，再扩平台数
- 先解决账号隔离、调度、审计、风控，再做商业化包装
- 先做团队可用，再做成熟产品

## 目标架构

建议采用“控制面 + 执行面”的结构。

### 接入层

- Web Console：运营后台
- Open API：对外集成
- CLI：保留 `sau` 作为本地和运维入口
- Skills / Agent Gateway：给大模型和 Agent 调用
- Webhook / Callback：任务状态回传

### 控制面

- 身份与权限：用户、组织、工作区、角色
- 账号中心：平台账号、会话、代理、健康状态
- 素材中心：视频、图片、封面、内容草稿
- 任务中心：发布计划、执行记录、失败重试
- AI 中台：标题、描述、标签、多平台改写
- 风控中心：规则、频控、异常识别、账号熔断
- 审计中心：操作审计、发布审计、模型审计

### 执行面

- 调度器：定时触发、补偿、重试
- 工作流编排器：长流程状态管理
- Worker 管理：把任务分发给执行节点
- Browser Worker：真正运行浏览器自动化的执行单元
- Platform Adapter：复用现有 `uploader/*`
- Credential Provider：运行时注入账号会话

### 数据面

- `PostgreSQL`：主业务库
- `Redis`：队列、锁、状态缓存
- 对象存储：素材、截图、回执、封面
- Secret Store / KMS：cookie、token、代理等敏感信息
- 日志与指标系统：平台运行与审计观察

## 推荐产品分阶段

### 第一阶段

目标：让你个人稳定使用，并把仓库整理成“可平台化”的结构。

这阶段不追求完整后台，而是先收敛主线。

应完成：

- 明确标记 legacy 路径
- 从 `sau_cli.py` 抽出统一应用服务层
- 给 `uploader/*` 增加稳定的 adapter 抽象
- 建立统一任务模型与错误码
- 建立基础账号中心和凭证存储策略
- 保证 CLI 仍然可用

这阶段交付的本质是：从“脚本入口”升级为“平台可调用内核”。

### 第二阶段

目标：让一个小团队能够在平台里协作使用。

建议 MVP 只覆盖四个主线平台：

- 抖音
- 快手
- 小红书
- Bilibili

这阶段应完成：

- 新控制面后端
- 新管理后台前端
- 租户、工作区、用户、角色
- 账号管理与状态检查
- 素材库、草稿库
- 发布计划与执行历史
- 调度器、Worker、基础重试
- 基础审计
- 基础风控

这阶段结束后，产品应该已经是“多人可用的平台”，而不是个人脚本管理台。

### 第三阶段

目标：从协作平台升级为可产品化交付的平台。

这阶段重点不再是“能发”，而是“能治理、能扩张、能接 AI”。

应完成：

- 完整多租户隔离
- 配额、权限和套餐预留
- AI 中台接入
- 审核流
- 更完整的风控策略
- 更强的可观测和告警
- CLI 与平台远程模式兼容
- Skills 与平台任务桥接

### 第四阶段

目标：从发布平台升级为自动运营平台。

这阶段重点是把内容生产、审核、发布、复盘串起来。

应完成：

- 选题到发布的内容流水线
- 多平台内容派生与改写
- 运营策略引擎
- Open API / Webhook
- 数据分析与复盘看板
- 评论、私信、再利用等 AI 辅助运营能力

## 核心模块规划

### 1. 执行内核

从 `sau_cli.py` 中提炼出统一服务层，避免未来出现 CLI 一套、API 一套、Worker 一套的三轨维护问题。

建议拆出：

- `PlatformAccountService`
- `PlatformPublishService`
- `PublishPlanService`
- `PlatformCapabilityService`

CLI 只负责参数解析和结果输出，真正的平台动作交给服务层。

### 2. 平台适配层

现有 `uploader/*` 不重写，而是统一包一层稳定接口。

建议接口包含：

- `login`
- `check`
- `upload_video`
- `upload_note`
- `refresh_session`
- `get_capabilities`

这样未来平台、CLI、skills 都能走同一协议。

### 3. 账号与凭证中心

这是从个人工具升级到多人平台最关键的部分。

必须做到：

- cookie / session 加密存储
- 执行时临时解密
- 账号会话按租户隔离
- 一账号默认单并发
- 账号状态、失效、风控命中要可追踪

### 4. 任务与调度中心

建议建立标准任务状态机：

- `draft`
- `risk_checking`
- `approval_pending`
- `scheduled`
- `dispatched`
- `running`
- `succeeded`
- `failed`
- `quarantined`
- `canceled`

这个模块要支持：

- 立即发布
- 定时发布
- 重试
- 补偿
- 幂等
- 失败归因

### 5. 素材与内容中心

建议把“素材”和“平台发布参数”拆开。

核心对象可以是：

- `Asset`
- `ContentDraft`
- `PublishVariant`
- `PublishPlan`
- `Template`
- `BrandProfile`

原则是一份源内容，多平台派生，不让前端直接拼平台底层字段。

### 6. AI 中台

AI 接入建议从辅助型能力切入：

- 标题生成
- 描述改写
- 标签推荐
- 多平台文案改写
- 敏感内容规避建议
- 发布时间建议

后续再扩展到：

- 评论/私信辅助
- 系列内容生成
- 品牌风格一致化
- 内容复盘与优化建议

AI 模块必须保留 trace，包括：

- 使用的模型
- prompt 模板版本
- 输入上下文
- 输出结果
- 规则校验结果

### 7. 风控中心

风控不能只是敏感词，它应该贯穿全流程。

应覆盖三层：

- 发布前：参数合法性、平台限制、敏感词、重复内容
- 执行中：识别登录失效、验证码、风控页、异常跳转
- 发布后：回查状态、统计失败率、账号健康评分

MVP 就应该有的规则：

- 一账号单并发
- 连续失败自动熔断
- 高频重复内容限制
- 失败原因分类
- 风控命中单独标记，不和普通失败混在一起

### 8. 审计与可观测

建议所有关键动作保留可追踪记录：

- 谁操作的
- 操作了哪个账号和任务
- AI 做了什么改写
- 风控命中了什么规则
- 最终成功还是失败
- 有哪些页面证据和日志

这会直接决定未来产品能不能给团队和客户放心使用。

## 平台风控专项建议

这部分需要特别重视，因为未来平台不是个人单号使用，而是多人、多账号、多任务同时运行。

### 账号层

- 一账号一浏览器上下文
- 一账号默认单并发
- 高风险账号进入隔离态
- 支持账号级代理与地域策略

### 内容层

- 检测素材哈希重复
- 避免短时间内多账号分发完全相同内容
- 平台文案自动做轻量差异化
- 高风险类目默认强制人审

### 任务层

- 设置冷却时间
- 控制租户级突发量
- 登录失效、验证码、限流作为特殊错误类型
- 避免无限重试

### AI 层

- AI 只先做建议稿
- 发布前必须过规则引擎
- 高风险租户或类目必须人审
- 模型输出必须可审计

## 你这条路上的真正护城河

如果这套平台做强，真正的价值不只是“能自动上传”，而是下面几件事组合起来：

- 真实可用的平台自动化适配器
- 多人、多账号、任务化的稳定执行体系
- AI 驱动的内容派生与运营协作
- 对各平台风控规律的长期积累
- 审计、治理和可追责能力

也就是说，真正的产品不是一个上传器，而是一套“自媒体自动运营执行中枢”。

## 主要风险与取舍

### 旧 Web 继续修补

这是最需要避免的路线。短期看快，长期会让系统更难维护。

### CLI、API、Worker 三套逻辑分叉

必须先抽统一服务层，不然以后修一个平台 bug 要改三处。

### 凭证安全

未来不能再按本地脚本思路存 cookie。必须上加密、隔离、临时注入和审计。

### 过早追求全自动 AI

AI 很重要，但早期应该先做“AI 助理”，不是“AI 全自动驾驶”。否则一旦误发、撞风控、触发违规，损失会很大。

### 一开始就做成重微服务

不建议。前期保持一个清晰的模块化单体或轻量服务体系更稳，等任务编排和租户规模起来后，再按瓶颈拆。

## 推荐执行顺序

1. 明确 legacy 边界
2. 抽取主线内核
3. 建立平台 adapter 抽象
4. 建立新任务模型和状态机
5. 落地账号中心与凭证安全
6. 做最小管理后台和调度器
7. 先跑通 4 个主线平台
8. 再接 AI 和审核流
9. 最后做更完整的运营自动化

## 当前建议

如果你现在就要启动这件事，最合适的第一步不是先画漂亮后台，而是先完成这三件事：

1. 把旧 Web 正式标记为 legacy
2. 把 `sau_cli.py` 背后的调用逻辑抽成可复用服务层
3. 定义未来平台的数据模型和任务状态机

只要这一步做好，后面的多人平台、AI、风控、审计，都会顺很多。
