# 多人平台实施任务拆分

## 说明

这份文档不是开发排期表，而是把演进过程拆成更容易推进的任务包。顺序遵循一个原则：

先收敛主线，再平台化；先解决多人协作，再做 AI 自动运营。

## 阶段一

### 目标

把当前仓库整理成“可被平台调用的执行内核”。

### 任务包

1. 标记 legacy 路径
   - 在文档中明确 `sau_backend.py`、`sau_frontend/`、`myUtils/` 的历史状态
   - 新增说明：未来平台不再基于旧 Web 继续开发

2. 抽取主线服务层
   - 从 `sau_cli.py` 中抽出账号服务、发布服务、平台能力服务
   - CLI 只保留参数解析和输出
   - 让未来 API 和 Worker 能复用同一套服务层

3. 建立平台 adapter 接口
   - 为 `uploader/*` 增加统一封装接口
   - 抽象 `login`、`check`、`upload_video`、`upload_note`
   - 定义平台能力声明，例如是否支持图文、是否支持定时发布

4. 统一错误模型
   - 定义登录失效、页面变更、平台限流、风控命中、参数非法、执行超时等错误码
   - 建立标准返回结构

5. 建立基础任务模型
   - 设计 `PublishPlan`
   - 设计 `Task`
   - 设计 `TaskRun`
   - 设计 `AccountSession`
   - 设计 `AuditEvent`

### 完成标志

- CLI 仍然能正常用
- 平台调用不再直接依赖 CLI 文件结构
- uploader 已经可以通过统一接口被调用

## 阶段二

### 目标

做出一个可供小团队使用的 MVP 平台。

### 范围

建议先只覆盖四个平台：

- 抖音
- 快手
- 小红书
- Bilibili

### 任务包

1. 新平台后端骨架
   - 用户、组织、工作区
   - 角色与权限
   - 账号、素材、草稿、任务、审计基础模块

2. 新平台前端骨架
   - 登录页
   - 工作区切换
   - 账号页
   - 素材页
   - 草稿页
   - 发布计划页
   - 任务历史页

3. 账号中心 MVP
   - 平台账号注册
   - 登录二维码中转
   - 会话状态检查
   - 账号健康状态展示

4. 素材与内容中心 MVP
   - 视频、图片上传
   - 草稿保存
   - 基础模板
   - 多平台元数据编辑

5. 调度与执行 MVP
   - 立即发布
   - 定时发布
   - Worker 执行
   - 基础重试
   - 基础失败归因

6. 审计与日志 MVP
   - 操作留痕
   - 发布留痕
   - 失败截图或回执记录

7. 风控 MVP
   - 一账号单并发
   - 敏感词检测
   - 重复内容限制
   - 失败自动熔断

### 完成标志

- 运营人员可以在后台管理账号、素材和发布计划
- 定时任务可稳定下发
- 失败可追踪，账号不会乱串

## 阶段三

### 目标

把 MVP 平台升级为具备产品化能力的平台。

### 任务包

1. 多租户隔离
   - `tenant_id`
   - `workspace_id`
   - 租户配置
   - 资源归属和权限边界

2. 凭证安全升级
   - cookie / session 加密存储
   - 临时解密执行
   - 敏感信息日志脱敏

3. 审核流
   - 提交审核
   - 审核通过后入调度
   - 审核意见留痕

4. AI 中台初版
   - 标题生成
   - 描述改写
   - 标签推荐
   - 多平台文案改写
   - 输出 trace 记录

5. 风控增强
   - 账号健康分
   - 登录异常识别
   - 验证码与风控页分类
   - 平台失败率统计

6. 可观测增强
   - 指标
   - 告警
   - Trace
   - Worker 健康监控

7. CLI 与平台联动
   - 保留本地模式
   - 增加远程提交模式
   - 让 `skills/*` 可以桥接平台任务

### 完成标志

- 平台可支持多个团队或客户独立使用
- AI 可以稳定参与内容生成与审核辅助
- 出问题时能快速定位原因

## 阶段四

### 目标

从发布平台升级为自动运营平台。

### 任务包

1. 内容流水线
   - 选题
   - 草稿
   - AI 派生
   - 风控校验
   - 审核
   - 调度发布
   - 发布后复盘

2. 运营策略引擎
   - 多账号矩阵编排
   - 多平台节奏控制
   - 系列内容策略
   - A/B 标题

3. 数据分析
   - 发布成功率
   - 平台失败率
   - 账号健康趋势
   - AI 采用率
   - 内容模板表现

4. 外部接入
   - Open API
   - Webhook
   - 第三方素材系统对接
   - Agent 自动运营接口

5. 深度 AI 场景
   - 评论建议
   - 私信草稿
   - 封面文案建议
   - 再利用内容生成

### 完成标志

- 平台从“自动发布工具”升级成“自动运营中枢”

## 贯穿所有阶段的约束

### 账号隔离

- 一账号一浏览器上下文
- 一账号默认单并发
- 高风险账号可单独隔离执行

### 风控优先

- 发布前校验
- 执行中识别异常页面
- 发布后回查与统计

### AI 默认辅助

- 默认先做人机协同
- 高风险内容必须可审
- 模型输出必须可追踪

### 避免三套逻辑分叉

- CLI、平台 API、Worker 必须共用统一服务层
- 平台不能绕开 adapter 直接写网页逻辑

## 建议的优先顺序

如果现在开始推进，建议优先做这几件事：

1. 明确 legacy 边界
2. 抽取主线服务层
3. 设计任务状态机与数据模型
4. 建立账号中心与凭证安全方案
5. 做四大主线平台的 MVP 调度闭环
6. 再接 AI 与审核流

## 暂时不要急着做的事情

- 不要继续扩旧 Web 功能
- 不要一开始就追求微服务大拆分
- 不要先做全平台接入
- 不要一开始就让 AI 全自动发布
- 不要把风控放到最后才补
